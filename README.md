## 4. Отсутствие `cache` для зависимостей

Каждый запуск заново скачивает зависимости, `pip` каждый раз тратит время на загрузку пакетов, увеличивается нагрузка на сеть и PyPI, pipeline замедляется.

Добавим параметры в `cache` и `variables`. Теперь зависимости кэшируются между запусками, повторные pipeline используют уже скачанные пакеты.

В итоге CI будет работать быстрее, особенно на больших проектах.

# Реализуем работу с секретами

Почему хранение секретов в CI/CD Variables — плохая практика

GitLab CI/CD Variables безопаснее, чем хранение секретов в коде, однако такой подход все равно имеет ряд недостатков:

1. Секреты хранятся внутри GitLab.
2. Разработчики с достаточными правами могут получить доступ к переменным.
3. Секреты сложно централизованно ротировать.
4. Нет полноценного контроля времени жизни секрета.
5. Секрет может случайно попасть в логи pipeline.
6. При компрометации GitLab могут утечь все секреты проекта.

7. Hashicorp Vault предоставляет:

- централизованное хранение секретов
- аудит доступа
- безопасную выдачу секретов
- ротацию секретов
- TTL (время жизни секретов)
- разграничение прав доступа
- возможность динамической генерации секретов

Vault используеv как внешнее хранилище секретов для CI/CD pipeline.

Создадим `Dockerfile`, чтобы у всех пользователей было одинаковое окружение, и `Docker Compose`, используюзий образ Hashicorp Vault, который позволяет безопасно хранить секреты. 
Запустим: `docker compose up -d`

После запуска контейнера Vault был открыт в браузере, для входа использовался токен root.

Далее был написан скрипт получения секрета `get_secret.sh`. Он указывает адрес Vault сервера, передает токен авторизации, отправляет HTTP-запрос в Vault API, запрос обращается к пути `/v1/kv/data/corgi` и получает значение секрета `fun_fact`. Из JSON ответа мы извлекаем только его значение и в итоге он сохраняется только как перменная окружения. 

В пайплайн пришлось добавить: 
```
before_script:
  - apt-get update && apt-get install -y curl jq
```

curд - для HTTP-запросов к Vault API, jq - для обработки JSON ответа от Vault.

А также в deploy stage: 
```
- chmod +x scripts/get_secret.sh
- ./scripts/get_secret.sh
```
чтобы сделать скрипт исполняемым и вообще запускать получение секрета. 

Однако оказалось, что что-то с безопасностью все-таки не так и мой пайплайн никак не хотел проходит этап security. 

<img width="930" height="366" alt="image" src="https://github.com/user-attachments/assets/5a5cadc1-32a1-4172-a808-4d3fc6da951a" />

Все решилось после замены 
```
security_scan:
  stage: security
  script:
    - bandit -r .
```

на 
```
security_scan:
  stage: security
  script:
    - bandit -r . -s B101,B311
```

Команда `- bandit -r .` сканировала весь проект. Проверка B101 связана с использованием assert в pytest-тестах, что является стандартной практикой для unit testing и не представляет угрозы безопасности. Проверка B311 относится к использованию random.choice(), но его мы использовали только для выбора фактов про корги и ничего не шифровали.
